# video-steganography-java

Hide and retrieve secret messages inside video files using LSB (Least Significant Bit) substitution. Built with Java and OpenCV — encrypts payloads with AES-GCM and distributes ciphertext bits across random locations derived from a passphrase.

---

## How It Works

### Encoding (Encoding.java)
1. The user enters a secret message
2. The user enters a passphrase
3. A random salt and nonce are generated and PBKDF2 derives a 256-bit AES key
4. The message is encrypted using **AES/GCM/NoPadding**
5. Ciphertext length, salt, and nonce are embedded into **frame 0**
6. A pseudorandom seed is derived from the passphrase (not stored in video)
7. Ciphertext bits are distributed across unique random locations (frame, row, column, channel)
8. The modified video is saved as `output.avi` using the **FFV1 lossless codec**

### Decoding (Decoding.java)
1. The user enters the same passphrase used for encoding
2. Frame 0 of `output.avi` is read to recover ciphertext length, salt, and nonce
3. The passphrase regenerates the identical random payload locations
4. Hidden ciphertext bits are read back from those locations
5. AES-GCM decrypts and authenticates the plaintext message

### Why LSB?
Changing the least significant bit of a colour channel shifts its value by at most 1 out of 255. This difference is completely imperceptible to the human eye, making the modification invisible.

```
Original pixel:   Blue=180  Green=202  Red=141
After embedding:  Blue=180  Green=203  Red=140   ← difference of 1, invisible
```

---

## Pixel Layout (Frame 0)

| Columns | Data stored |
|---------|-------------|
| Start   | 32-bit ciphertext length |
| Next    | 16-byte PBKDF2 salt |
| Next    | 12-byte AES-GCM nonce |
| Rest    | unused |

## Prerequisites

### Sender machine
- Java JDK 11+
- `opencv-4120.jar`
- `opencv_java4120.dll`
- `opencv_videoio_ffmpeg4120_64.dll`
- `input.mp4` — the cover video

### Receiver machine
- Java JRE/JDK 11+
- `opencv-4120.jar`
- `opencv_java4120.dll`
- `opencv_videoio_ffmpeg4120_64.dll`
- `output.avi` — the encoded video received from sender

> The two `.dll` files are self-contained. No OpenCV installation is required on the receiver machine.

---

## Setup

1. Download the official OpenCV 4.12.0 Windows release from
   https://github.com/opencv/opencv/releases/tag/4.12.0

2. Copy these three files into your project folder:
   ```
   build\java\opencv-4120.jar
   build\java\x64\opencv_java4120.dll
   build\bin\opencv_videoio_ffmpeg4120_64.dll
   ```

3. Place your cover video in the project folder as `input.mp4`

## Usage

### Compile
```bash
javac -cp ".;opencv-4120.jar" Encoding.java
javac -cp ".;opencv-4120.jar" Decoding.java
```

### Encode a message
```bash
java --enable-native-access=ALL-UNNAMED -cp ".;opencv-4120.jar" -Djava.library.path=. Encoding
```
```
Enter secret message: Hello
Enter passphrase: mySecretPass
Video: 1280x534 @ 23.97fps, 329 frames
Using codec: FFV1
Done. Encoded 329 frames into output.avi
```

### Decode the message
```bash
java --enable-native-access=ALL-UNNAMED -cp ".;opencv-4120.jar" -Djava.library.path=. Decoding
```
```
Enter passphrase: mySecretPass
Opened output.avi: 329 frames
Cipher bytes:     21
Hidden message: Hello
```

## Sending the encoded video

The only file that needs to be transferred to the receiver is `output.avi`. The receiver must know the same passphrase used by the sender.

```
Sender  →  uploads output.avi to Google Drive  →  Receiver downloads and runs Decoding
```

> Make sure the file is transferred without re-encoding. Google Drive stores it as-is when uploaded as a generic file.

## Important Notes

- **FFV1 lossless codec is required.** Any lossy codec (MP4, XVID, H.264) will corrupt the hidden bits during compression and decoding will fail.
- Payload bits are spread across random non-zero frames and random pixel channels; frame 0 contains only non-secret metadata (cipher length, salt, nonce).
- Random placement seed is derived from passphrase and is never stored in the video.
- AES-GCM provides confidentiality and integrity (tampered video or wrong passphrase fails authentication).
- UTF-8 text is supported.
